# CLAUDE.md — ngj_10

Game jam project. Unity 2D. Repo: `damarrus/ngj_10`.

## Философия

Геймджем. Скорость и играбельность важнее чистоты. Цель — рабочий билд в срок, не идеальная архитектура.

- **Не оверинженерить.** Никаких DI-контейнеров, event-bus, абстрактных фабрик, ECS, паттернов ради паттернов. Прямой код, который видно насквозь.
- **Не говнокодить.** Понятные имена, мелкие методы, никаких `Update()` с 200 строк, никаких magic numbers без причины. Код должен читаться без расшифровки.
- **Баланс:** если решение занимает 10 строк просто или 100 строк «правильно» — берём 10, если они не выстрелят в ногу.
- **Дублирование терпимо** в малых дозах. Не плодим абстракцию из-за двух похожих кусков. Третий раз — рефакторим.
- **YAGNI жёстко.** Не пишем под «вдруг понадобится». Только то, что нужно сейчас.

## Тех-стек

- Unity **6000.4.9f1**, 2D (URP), Universal Render Pipeline 17.4
- **Целевая платформа — WebGL** (веб-билд в первую очередь). Всё писать с оглядкой на браузер.
- Input System (new) + legacy (Active Input Handling = **Both**)
- C#, namespace `Ngj10` (asmdef `Ngj10` в `Assets/_Project/Scripts/`)

## WebGL — главный таргет

Игра в первую очередь для веб-билда. Ограничения браузера — учитывать всегда:

- **Нет многопоточности по умолчанию.** Не использовать `System.Threading.Thread`, тяжёлый `Task.Run`. Корутины / async-await на main thread — ок.
- **Файловая система ограничена.** Нет прямого `System.IO` к произвольным путям. Сохранения — через `PlayerPrefs` (он мапится в IndexedDB) или `Application.persistentDataPath`. Не читать/писать произвольные файлы.
- **Звук:** WebAudio. Первый звук может требовать user-gesture (клик) — браузеры блокируют автоплей. Не рассчитывать на звук до первого инпута.
- **Размер билда критичен** — грузится по сети. Не тащить тяжёлые пакеты, большие текстуры, лишние зависимости. Сжимать спрайты.
- **Производительность:** WebGL медленнее нативного. Минимум аллокаций в `Update`, не спавнить тысячи объектов, пулить если много.
- **Нет:** рефлексии-хеви кода (IL2CPP стрипает), `Process`, серверных сокетов. Сеть — только `UnityWebRequest` / fetch.
- **Ввод:** мышь + клавиатура + тач. Геймпад в браузере ненадёжен — не основной.
- **Курсор/фуллскрин** — по user-gesture, не программно в любой момент.

Билд: `File > Build Profiles`, платформа **Web**. Проверять в браузере, не только в Editor (поведение отличается).

## Структура

```
Assets/_Project/
  Scenes/      Boot, Game, End  (build order: Boot=0 -> Game=1 -> End=2)
  Scripts/
    Core/      GameManager, SceneLoader, BootStarter, GameState, SceneNames
    Gameplay/  игровая логика
  Prefabs/  Art/{Sprites,UI}  Audio/{Music,SFX}  Materials/  Settings/
```

Весь новый контент — под `Assets/_Project/`. Не сорить в корень `Assets/`.

## Поток сцен

- **Boot** — точка входа. `BootStarter` создаёт `GameManager` (persistent), авто-грузит Game.
- **Game** — геймплей.
- **End** — экран конца, рестарт по любой клавише.
- Переходы только через `SceneLoader` / `GameManager`. Имена сцен — в `SceneNames`, не хардкодить строки.

## Код-конвенции

- `PascalCase` классы/методы/публичные поля, `_camelCase` приватные поля, `[SerializeField] private` для инспектора.
- Один класс — один файл, имя файла = имя класса.
- Namespace по папке: `Ngj10.Core`, `Ngj10.Gameplay`.
- Комментарии — только где «почему», не «что». Очевидное не комментировать.
- Magic numbers геймплея → `[SerializeField]` поля, чтобы тюнить в инспекторе без рекомпиляции.

## Unity MCP — как я работаю с Editor

Подключён mcp-unity (инструменты `mcp__mcp-unity__*`): сцены, GameObject, компоненты, консоль.

- **Фокус-throttle:** Unity тормозит мост когда окно не в фокусе → MCP-запросы таймаутят. Перед серией MCP-команд проси у оператора **один клик в Unity**. Это известное ограничение, конфигом не лечится.
- **Рекомпиляция:** новый/изменённый `.cs` компилится только когда Editor в фокусе (или через `recompile_scripts`, который рвёт WS — норма). После правки скрипта — фокус на Unity.
- **`.unity`/`.prefab`/`.asset` не редактировать руками** — сложный YAML + GUID-связи. Только через MCP.
- **Скрипты, ProjectSettings, JSON, .md** — можно править файлами напрямую.
- После правок проверять консоль (`get_console_logs` errors) на ошибки компиляции.
- **Ссылки в инспекторе через MCP ненадёжны — не связывать через MCP.** На компоненты (`Text`, MonoBehaviour) остаются `null`; на GameObject резолвятся на НЕ ТОТ объект (в ответе видно имя `"New Game Object"` вместо реального) → NRE в рантайме. Обход: находить всё кодом — дети по `transform.Find("Child")` в `Awake`, зависимости через `GetComponent<T>()` / `FindAnyObjectByType<T>()`. Не вешать `[SerializeField]`-ссылки в расчёте связать их через MCP.
- **`execute_menu_item` ненадёжен** — UI собирать вручную (`update_gameobject` + `update_component`), не через `GameObject/UI/*`. Play через MCP не запускается — тест в Play Mode руками оператором.
- **Unity 6 Find API:** `FindAnyObjectByType<T>()` (и `FindObjectOfType`, и `FindFirstObjectByType` — deprecated).
- Полные грабли и приёмы по MCP → `docs/unity-mcp-notes.md`. Дополнять при новых открытиях.
