# UI-конвенции и грабли верстки

Выстрадано на лидерборде (2026-06-07). Кратко: **UI собираем на сцене руками, код
держит только логику**. Ниже — почему и какие грабли набили.

## Главное правило: UI НЕ генерим кодом

Интерфейсы строятся **на сцене** (объекты + RectTransform верстаются в редакторе,
подгоняются под макет). MonoBehaviour-код держит **только логику** + `[SerializeField]`
ссылки на готовые scene-объекты.

**НЕ делать:** построение UI в `Awake`/builder-скрипте (`AddComponent<Image>()`,
расстановка `anchorMin/Max`/`offsetMin/Max` в рантайме, спавн TMP кодом).

**Почему code-driven UI — антипаттерн здесь:**
- Нельзя крутить в Play Mode — каждая правка размера/шрифта = рекомпиляция.
- Тонкая верстка (отступы, выравнивание, подгон под макет) кодом мучительна:
  не видишь результат, итерируешь вслепую через rebuild.
- Скрин Game-View в Edit-mode без фокуса отдаёт **мёртвый кадр** (см. ниже) — нечем
  проверить результат правки.
- На лидерборде сожгли кучу итераций, пока не перешли на scene-objects.

**Как правильно:**
- Компонент: логика + `[SerializeField]` ссылки (panel, content, input, button,
  row-prefab). Ссылки заводятся в инспекторе (MCP wiring работает, см.
  `unity-mcp-notes.md`).
- Повторяющиеся элементы (строки списка) — `Instantiate` готового **scene/prefab
  шаблона** (неактивный объект на сцене), не построение элемента кодом. Шаблон
  верстают на сцене. После `Instantiate` — `SetActive(true)` на клоне (шаблон
  неактивен). Образец паттерна: `LeaderboardRow`/`LeaderboardView` — код только
  `Set(...)`/спавн, ноль вёрстки.
- Editor-builder-скрипт (генератор UI) — допустим как разовый костыль «накидать
  базовую иерархию», но **удалять сразу после**. Иначе при повторных прогонах
  плодит дубли (см. ниже) и перетирает ручную верстку.

Роли: ассистент накидывает базу (объекты + скрипты-логику), оператор/редактор
доводит верстку на сцене.

### Если правка RectTransform всё же идёт кодом/через MCP

Вёрстку держим на сцене (выше). Но когда точечно двигаешь rect через MCP
(`gameobject-component-modify` на RectTransform) — порядок и нюансы, чтобы не
итерировать вслепую:

- **Порядок свойств:** `anchorMin` → `anchorMax` → `pivot` → `anchoredPosition` →
  `sizeDelta`. `anchoredPosition`/`sizeDelta` считаются ОТНОСИТЕЛЬНО текущих anchor —
  поставишь позицию до anchor, элемент прыгнет.
- **Center-anchor** (`anchorMin=anchorMax=pivot=0.5`): `sizeDelta` = реальный
  размер. Самый предсказуемый случай.
- **Split/stretch anchor** (`anchorMin≠anchorMax`): `sizeDelta` УЖЕ не ширина/высота,
  а offset поверх anchor-rect — гнать `offsetMin`/`offsetMax`. Это ровно случай
  Content flush к Viewport ниже (`offsetMin.x=offsetMax.x=0`, anchor stretch по X).
- **Инвертированные знаки:** Inspector Top=10 → `offsetMax.y=-10`; Right=10 →
  `offsetMax.x=-10`.
- **НИКОГДА не читать размер layout в том же кадре правки.** LayoutGroup /
  ContentSizeFitter пересчитываются на СЛЕДУЮЩЕМ кадре — `sizeDelta`/rect сразу
  после правки стейл → «чинишь» несуществующий баг. Нужен размер сейчас:
  `Canvas.ForceUpdateCanvases()` + `LayoutRebuilder.ForceRebuildLayoutImmediate(rect)`,
  затем читать. (Через MCP — лучше дождаться кадра / Play, см. скрин ниже.)

## Грабли верстки (UGUI / ScrollRect)

### Content шире viewport → правый столбец за маской

**Симптом:** в строках ScrollRect-списка правый столбец (счёт) обрезан правым краем,
строки уходят за границу панели; левый край тоже подрезан.

**Корень:** дочерний `Content` шире `Viewport` (ненулевой горизонтальный offset или
`sizeDelta.x`). `RectMask2D` на Viewport режет всё, что вылезло. У нас Content имел
L/R offset ~−50 → контент уехал за маску.

**Фикс:** держать горизонтальные края Content **flush** к Viewport:
`offsetMin.x = offsetMax.x = 0`, `sizeDelta.x = 0`, anchor stretch по X (0..1).
Строки (Row) тогда = ширине Content = ширине Viewport, столбцы внутри видимы.

**Диагностика:** если правки rect внутреннего элемента (сдвиг столбца) не двигают
картинку — проблема не в элементе, а в том что весь Row/Content за маской. Сдвинуть
подозрительный элемент сильно (на 100+px) — если не появился, виноват родитель.

### Дубли при повторном запуске builder-скрипта

Editor-builder, создающий объекты, при каждом прогоне плодит копии (`Build()` ×8 →
8 шаблонов строк под Canvas). Если уж генеришь — в начале **удаляй прежние** по имени
(`transform.Find` + `DestroyImmediate`), идемпотентно. Лучше — удали builder после
первой генерации.

Чистка дублей пост-фактум: найти все по имени, оставить тот, на который ссылается
рабочий компонент (`SerializedObject.FindProperty("_rowPrefab").objectReferenceValue`),
остальные `DestroyImmediate`.

## Скриншот для проверки верстки

- **Game View в Play Mode** — рендерит живой кадр, MCP-скрин свежий. Надёжно.
- **Game View в Edit Mode без фокуса окна** — MCP `screenshot-game-view` отдаёт
  **закешированный/мёртвый кадр** ИЛИ "render texture not available" (вкладка Game
  закрыта). Правки на сцене реальны, но скрин их НЕ показывает → легко уйти в ложные
  выводы. Не доверять Edit-mode Game-View скрину.
- **Scene View** — зависит куда смотрит редактор-камера (может быть пустота). Тоже
  ненадёжен для проверки UI.
- **Канон:** либо Play Mode + MCP-скрин, либо просить оператора скинуть скрин Game
  View (его клиент рендерит живьё). На лидерборде живые скрины оператора были
  единственным надёжным источником правды.

## TMP / общее

- Весь UI-текст — `TextMeshProUGUI` (см. CLAUDE.md). Legacy `UI.Text` запрещён.
- `enableWordWrapping` deprecated → `textWrappingMode = TextWrappingModes.NoWrap`.
- Оверлей «поверх всего» (борд над game-over) — отдельный `Canvas` на панели с
  `overrideSorting = true` + `sortingOrder` выше + свой `GraphicRaycaster`. Надёжнее
  чем `SetAsLastSibling` (не зависит от порядка в иерархии).
